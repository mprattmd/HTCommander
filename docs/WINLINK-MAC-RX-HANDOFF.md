# Winlink on macOS — RESOLVED (debugging handoff, kept for history)

**Status: ✅ RESOLVED — shipped in v0.4.7 (2026-06-05).** Winlink on macOS now connects,
receives, sends, and **delivers end-to-end** to a real inbox. Three fixes landed on `main`:

1. **macOS connected-mode RX** — `1b26dde` (`RadioController.cs`). The one-shot
   `REGISTER_NOTIFICATION` sent in `HandleDevInfo` silently doesn't take on IOBluetooth, so the
   radio pushed **zero** async events (HtStatus, Position, and the DATA_RXD packets carrying the
   peer's reply) → connect timed out. Fix: the poll loop re-sends `REGISTER_NOTIFICATION`
   (HtStatus+Position) until an async `EVENT_NOTIFICATION` confirms it took (`eventChannelLive`),
   then stops. The radio does **not** `cmd=6`-ACK these, so *receipt of an event* is the stop
   signal. Registering HtStatus+Position is enough — DATA_RXD then flows implicitly (do NOT
   register DataRxd). No-op on Linux (an event arrives on cycle 1). **The IOBluetooth-RFCOMM-receive
   theory below was WRONG** — the transport was fine; the subscription just wasn't sticking.
2. **Mail never delivered** — `dc6158c` (`WinLinkMail.cs` `SerializeMail`). Headers were built with
   `StringBuilder.AppendLine` = `\n` on macOS/Linux, but Winlink B2F needs **CRLF** (its own
   `FindFirstDoubleNewline` scans for `\r\n\r\n`). The CMS ACKed the proposal over RF then silently
   dropped the unparseable message → "Sent but never arrives." **Cross-platform bug (Linux too);**
   WinForms delivers only because `AppendLine` is `\r\n` on Windows. Fix: explicit `\r\n` per header.
3. **Radio not returning to operator's channel after a session** — `55d8a56` (`RadioController.cs`
   `OnSetLock`). Lock saved the return-to channel from stale `ActiveChannelA` (cached READ_SETTINGS,
   not updated on knob-turn). Fix: capture live `lastStatus.curr_ch_id`, mirroring the region.

**Key lesson:** the AX.25 "link instability" (CONNECTED↔CONNECTING flaps, DM resets, mail-body
desync) was an **artifact of the `HTCOMMANDER_REREGISTER` diagnostic build** flooding the BT link
every 1.5 s + firing the status-re-kick race — NOT a real AX.25 bug. The production build (stops
re-registering after the first event) completes the multi-fragment B2F exchange cleanly. Don't debug
AX.25 timing against a noisy diagnostic build.

The original investigation below is **kept for history** — note its leading hypothesis (IOBluetooth
RFCOMM receive gap) turned out to be wrong; the real cause was the missing notification subscription.

## Original investigation (HISTORICAL — hypothesis was wrong, see above)

Ran a full Winlink connect+send on the working **Linux** box, same radio / 145.05 / WC4EOC-8.
At the time this looked like a Mac IOBluetooth RX-path bug:

- Linux **receives `EVENT_NOTIFICATION: type=2` (DATA_RXD) — 9× in one session**, including the
  gateway's UA that completes the connect. Mac showed **0× type=2** on identical radio/RF/gateway.
- Event-type tally (Linux): `9× type=2`, `51× type=1`, `6× type=6`, `1× type=13`.

The Mac got **0** async events of any type because its subscription never registered — fixed by
re-registering until an event confirms it (fix #1 above), not by changing `htbt.swift`.

## One-line summary

Winlink connected-mode works on **Linux** but not **macOS** with the **same radio, same
frequency (145.05), same gateway (WC4EOC-8), same RF**. TX is byte-for-byte identical on both
platforms; the difference is that **inbound packet data (`DATA_RXD`, event type 2) never reaches
the Mac over IOBluetooth**, so the gateway's UA reply is never seen and the AX.25 connect times out.

## Symptom

- **Linux:** 1 SABM burst → UA comes back → connected.
- **macOS:** SABM burst repeats ~3–4× on the 11 s AX.25 `Connect` (T1) timer → no reply → `DISCONNECTED`.
  - The multiple bursts are **normal AX.25 retry**, *not* a triple-send bug. Confirmed in the log:
    one `ConnectEx`→`EmitPacket`→`SendPacket` per `Timer - Connect` firing, spaced by the 11030 ms timer.

## What is CONFIRMED

1. **Bluetooth/GAIA control link is healthy on Mac.** Connects, GAIA channel validated, continuous
   status frames (`cmd=13` READ_RF_CHANNEL, `cmd=20` GET_HT_STATUS), dev-info, settings all flow.
2. **TX is correct and identical to Linux.** SABM frame = `AE86688A9E86F0968A6882B0AE613F` (15 B),
   generated by shared `AX25Session`/`WinlinkClient`. `HT_SEND_DATA` returns `err=0`. The radio
   reports the correct channel on the bursts that matter: `tx=145.0500 FM WIDE HIGH`.
   The radio builds the over-the-air signal from the `HT_SEND_DATA` payload + current channel and is
   stack-agnostic, so the emitted RF is the same as Linux's.
3. **No `DATA_RXD` ever arrives on Mac — proven at the raw Bluetooth byte level.** Added raw-hex
   logging in `libhtbt`'s RFCOMM receive callback (see Instrumentation). Across a full session the
   only inbound frames are solicited status polls + `EVENT_NOTIFICATION: type=6`
   (HT_SETTINGS_CHANGED — these are echoes of *our own* channel-lock settings writes). **Zero
   `type=2` (DATA_RXD) bytes.** So it is not a decode bug and not RF — the bytes never arrive.
4. **Channel selection is NOT the cause.** The radio reassigns its GAIA RFCOMM channel between
   sessions (seen on ch 1 and ch 4); Mac probes SDP-advertised channels, Linux scans 1..30 and
   takes the first GAIA responder. Both land on the radio's GAIA/SPP channel. Failure reproduces
   regardless of which channel is selected.
5. **Voice/audio channel is OFF** during the failures (it's on-demand, default off), so it is not
   disrupting the hardware TNC.
6. **RF/range ruled out** by the user: latest build connects fine on Linux with the identical
   frequency and station.

## Leading hypothesis

The radio receives the gateway's UA but **does not push `DATA_RXD` over the IOBluetooth RFCOMM
link**, whereas it does over the Linux/BlueZ kernel RFCOMM socket. The radio can only distinguish
the two connections by RFCOMM-level properties (credit-based flow control / modem-status signals /
MTU), since the GAIA command sequence is shared code and identical. Suspect the bridge's RFCOMM
channel open/servicing in `libhtbt` (`htbt.swift`) vs the raw kernel socket on Linux.

## The ONE test that settles the remaining fork

Two possibilities remain: (A) gateway replies but radio won't forward over IOBluetooth [RX bug], or
(B) gateway never hears Mac's burst [TX/RF]. Distinguish with a **receive test independent of our
own TX**:

- **Option 1 (easiest, pure software A/B):** On the working **Linux** box, run one Winlink connect
  with debug logging and confirm an `EVENT_NOTIFICATION: type=2` appears right after burst 1. Mac
  never shows `type=2` on the same radio/RF → definitively the Mac IOBluetooth receive path.
- **Option 2 (decisive on RX):** Put the **Mac** radio on **144.390 (APRS)** or any frequency with
  packet traffic for ~60 s. If `type=2` appears → Mac RX works → chase the TX/region angle instead;
  if nothing appears even from a known transmitter → IOBluetooth RFCOMM receive path is the bug.

## Secondary lead (TX side, lower priority)

Channel-lock **region thrash** on Mac: log shows `Radio locked … region 3` then
`SendPacket … radio-now region=0`, and across two attempts it locked region 3 then region 0.
Burst 1 went out on `region 3` with the bank not loaded (`freq unknown`) — possibly the wrong
frequency — before settling on `region 0` = Winlink 145.05. The lock logic is shared with Linux, so
verify whether Linux thrashes too. Probably not the root cause (correct-freq bursts also got no
reply), but worth confirming. Region/target computation: `RadioController.OnSetLock` /
`WriteLockSettings` / `SetRegion` in `cross/HTCommander.Core/radio/RadioController.cs`.

## How to build & run (macOS)

```bash
# .NET 9 lives at ~/.dotnet
cd /Users/prattmd/ClaudeProjects/htcomm

# Rebuild the IOBluetooth bridge after editing htbt.swift:
bash mac/htbt/build.sh      # → cross/HTCommander.Platform.Mac/runtimes/osx/native/libhtbt.dylib

# Build + run the app with full Bluetooth + bridge debug logging:
~/.dotnet/dotnet build cross/HTCommander.UI.Avalonia/HTCommander.UI.Avalonia.csproj -c Debug
HTBT_DEBUG=1 ~/.dotnet/dotnet run \
  --project cross/HTCommander.UI.Avalonia/HTCommander.UI.Avalonia.csproj -c Debug --no-build \
  > /tmp/htcomm-raw.log 2>&1 &
```

### Useful greps on the log

```bash
# Bridge channel selection / SDP:
grep -E "SDP service|probing channels|GAIA validated" /tmp/htcomm-raw.log

# Inbound event types (looking for type=2 = DATA_RXD):
grep -oE "EVENT_NOTIFICATION: type=[0-9]+" /tmp/htcomm-raw.log | sort | uniq -c

# Raw inbound bytes at the Bluetooth layer (proves what the radio pushes):
grep "RX RFCOMM" /tmp/htcomm-raw.log

# TX bursts + AX.25 retry timing:
grep -E "Radio locked|SendPacket: 15B|HT_SEND_DATA response|Timer - Connect|ConnectEx|state changed" /tmp/htcomm-raw.log
```

## Key files

- `mac/htbt/htbt.swift` — IOBluetooth RFCOMM bridge (the suspect). `rfcommChannelData` = RX path.
- `cross/HTCommander.Platform.Mac/MacRadioTransport.cs` — C# side: GAIA deframing + callbacks.
- `cross/HTCommander.Platform.Linux/RadioBluetoothLinux.cs` — the working reference (BlueZ socket).
- `cross/HTCommander.Core/radio/RadioController.cs` — shared RX dispatch (`OnReceivedData`,
  `HandleDataReceived`), TX (`SendPacket`), channel lock (`OnSetLock`/`WriteLockSettings`).
  `NotifyDataRxd = 2` is the event we never receive on Mac.
- `cross/HTCommander.Core/winlink/WinlinkClient.cs`, `radio/AX25Session.cs` — shared connect logic.
